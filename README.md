# MAX-бот · Запись на мероприятия

Сервис для записи абитуриентов на университетские мероприятия — дни открытых дверей, пробные занятия, консультации, экскурсии. Записаться можно прямо в мессенджере [MAX](https://max.ru), управление событиями идёт через веб-админку с разделением ролей. Допуск на мероприятие — по QR-коду (оффлайн) или ссылке (онлайн).

> Команда «Лимон» · Хакатон «Весенний код» · РТУ МИРЭА

## Что умеет

### Для абитуриента (в боте MAX)
- 👋 Согласие на обработку ПД при первом запуске
- 📅 Каталог ближайших мероприятий с подсчётом свободных мест в реальном времени
- ✍️ Запись в один тап, отмена до начала мероприятия
- 🎫 Уникальный QR-код для оффлайн-мероприятий (HMAC-подпись, подделать без секрета нельзя)
- 🔗 Прямая ссылка-кнопка для онлайн-мероприятий
- 🔔 Включение/выключение push-уведомлений по каждой записи
- 🎫 Раздел «Мои записи» с возможностью пересмотреть QR/ссылку

### Для организатора (веб-админка)
- 📝 Создание, редактирование и удаление мероприятий
- 👥 Списки участников с фильтрацией по статусу (записан / отменил / пришёл)
- 🔔 Push-рассылка участникам: перенос времени, изменение места, напоминания за 24ч и 1ч, информационные сообщения
- 📊 Статистика записей и посещаемости
- 📥 Загрузка списка контролёров из XLSX
- 📤 Выгрузка списка участников в XLSX
- 🔒 Закрытие/открытие регистрации, удаление мероприятия

### Для контролёра (на входе)
- 📱 QR-сканер через камеру смартфона прямо в браузере
- ✅ Мгновенная проверка валидности кода и допуск участника
- 🚫 Защита от повторных проходов и QR от других мероприятий

### Для техадмина
- 👤 Управление всеми учётками (организаторы, контролёры, другие техадмины)
- ✏️ Редактирование ФИО, смена паролей, отключение/активация учёток
- 📜 Полный журнал аудита всех значимых действий в системе
- 👀 Просмотр всех мероприятий для диагностики

│  │ Обработка  │  │ Дашборд, события,   │ │
│  │ событий    │  │ участники, аудит,   │ │
│  │ бота       │  │ QR-сканер, экспорт  │ │
│  └────────────┘  └─────────────────────┘ │
│         │                  │             │
│         └─────────┬────────┘             │
│                   ▼                      │
│        ┌──────────────────────┐          │
│        │   Repositories       │          │
│        │   (единый слой       │          │
│        │   доступа к БД)      │          │
│        └──────────┬───────────┘          │
└───────────────────┼──────────────────────┘
                    ▼
            ┌───────────────┐

            │  7 таблиц     │
            │  + audit_log  │
            └───────────────┘

**Принцип:** все обращения к БД идут через репозитории (`app/repositories/repos.py`) — это единственное место в коде, которое знает про SQL. Это даёт единую точку для аудита и фильтрации.

### Роли и доступ

| Роль | Где живёт | Что видит | Что может |
|---|---|---|---|
| **Абитуриент** | Бот MAX | Каталог открытых мероприятий, свои записи | Записаться, отменить, переключить уведомления |
| **Организатор** | `/admin` | Свои мероприятия | Создавать события, рассылать push, выгружать списки, назначать контролёров |
| **Контролёр** | `/admin` | Назначенные ему мероприятия (только QR-сканер) | Сканировать QR и отмечать пришедших |
| **Техадмин** | `/admin` | Всё | Создавать учётки, видеть аудит, диагностика |

## Подробная установка

### 1. Python и виртуальное окружение

Нужен Python 3.12 с [python.org](https://www.python.org/downloads/windows/) (не Microsoft Store).

> ⚠️ На Windows проверьте, какой `python` подхватился: `where.exe python`. Если первым идёт MSYS2 (`C:\msys64\...`), используйте `py -3.12` вместо `python` для создания venv — иначе venv создастся в Unix-стиле и не будет работать.

```powershell
py -3.12 -m venv .venv
.\.venv\Scripts\Activate.ps1
```

Если PowerShell ругается на политику выполнения:
```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
```

### 2. Зависимости

```bash
pip install -r requirements.txt
```

### 3. PostgreSQL

Локально через Docker — самый быстрый способ:

```bash
docker run -d --name maxbot-pg \
  -e POSTGRES_USER=maxbot \
  -e POSTGRES_PASSWORD=maxbot \
  -e POSTGRES_DB=maxbot \
  -p 5432:5432 \
  postgres:16
```

Или установить локально и создать БД:
```sql
CREATE USER maxbot WITH PASSWORD 'maxbot';
CREATE DATABASE maxbot OWNER maxbot;
```

## Настройка `.env`

Скопируйте `.env.example` в `.env` и заполните:

```env
# MAX Bot API
MAX_BOT_TOKEN=<токен от @MasterBot>
MAX_API_BASE=https://platform-api.max.ru

# Публичный URL для webhook (HTTPS обязательно для production)
PUBLIC_BASE_URL=https://your-domain.example.com

# Случайные секреты (32+ символа каждый, все три разные)
WEBHOOK_SECRET=<случайная строка для подписи webhook MAX>
QR_SECRET=<случайная строка для HMAC QR-токенов>
FLASK_SECRET_KEY=<случайная строка для cookie Flask-сессии>

# База данных
DATABASE_URL=postgresql://maxbot:maxbot@localhost:5432/maxbot

LOG_LEVEL=INFO
PORT=8080
```

Сгенерировать секреты:
```bash
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

> Локально без MAX-бота можно поставить заглушку: `MAX_BOT_TOKEN=stub`, `PUBLIC_BASE_URL=http://localhost:8080` — веб-админка будет работать, бот — нет.

## Подключение MAX-бота

### Получение токена

1. В мессенджере MAX найдите [@MasterBot](https://max.ru/masterbot)
2. Отправьте команду `/create`, придумайте имя и юзернейм
3. Получите `MAX_BOT_TOKEN` — впишите в `.env`

### Регистрация webhook

Бот должен быть доступен по HTTPS. Локально проще всего через [ngrok](https://ngrok.com):

```bash
ngrok http 8080
```

В выводе будет ссылка вроде `https://abc-123.ngrok-free.app` — её в `PUBLIC_BASE_URL`.

Затем регистрация webhook:
```bash
python -m migrations.register_webhook
```

Этот скрипт вызовет `POST /subscriptions` у MAX API и подпишет бота на события:
- `message_created` — сообщение пользователю
- `bot_started` — первый запуск бота
- `message_callback` — нажатие на inline-кнопку

## Создание учёток

Первого техадмина создайте через миграцию:

```bash
python -m migrations.create_users
```

Скрипт спросит email, ФИО и пароль. Дальше все учётки (организаторы, контролёры, техадмины) создаются через веб-админку: `/admin/users` под техадмином.

> 🔒 Учётки создаёт **только** техадмин. Организатор не может создать себе или коллеге учётку даже на загрузке контролёров из XLSX — только привязать уже существующих.

## БД и миграции

При первом запуске:
```bash
python -m migrations.init_db        # создать все таблицы
python -m migrations.create_users   # создать первого админа
```

Если разворачивали раньше без таблицы контролёров:
```bash
python -m migrations.add_controllers   # идемпотентно, можно повторно
```

Демо-данные для проверки (создаст 3 тестовых мероприятия от первого организатора):
```bash
python -m migrations.seed_events
```

### Схема БД

```
users               — абитуриенты (только max_user_id + username + согласие)
organizers          — сотрудники приёмной комиссии (email + bcrypt-пароль)
controllers         — контролёры на входе мероприятий
admins              — технические администраторы
events              — мероприятия
event_controllers   — N:M связь: какие контролёры допущены к мероприятию
registrations       — записи на мероприятия (с QR-токеном и кодом)
notifications       — журнал push-рассылок
audit_log           — все значимые действия в системе
```

Особенности схемы:
- **Partial unique index** на `registrations(user_id, event_id) WHERE status='confirmed'` — гарантирует, что один пользователь не может иметь две активные записи на одно мероприятие. После отмены — может записаться повторно.
- **CHECK-constraints** на `duration_minutes > 0` и `max_participants > 0`.
- **Foreign keys** `ON DELETE RESTRICT` для events — нельзя удалить мероприятие, на которое уже есть записи.

## Развёртывание на Railway

1. Создайте новый проект из GitHub-репозитория
2. Добавьте сервис **PostgreSQL** — Railway автоматически прокинет `DATABASE_URL`
3. В переменных проекта заполните остальные секреты из `.env`
4. После первого деплоя выполните миграции через Railway CLI:

```bash
railway run python -m migrations.init_db
railway run python -m migrations.create_users
```

5. Возьмите публичный URL проекта (например, `https://maxbot-production-XXXX.up.railway.app`) и:
   - Пропишите в переменной `PUBLIC_BASE_URL`
   - Зарегистрируйте webhook: `railway run python -m migrations.register_webhook`

Railway раздаёт URL вида `postgres://...`, но SQLAlchemy 2.x ждёт `postgresql://...` — в коде это нормализуется автоматически (см. `app/db.py`).

ProxyFix в `app/main.py` доверяет заголовкам `X-Forwarded-*` — нужно для корректной работы за reverse-proxy Railway.

## Использование

### Сценарий «Абитуриент → Запись»

1. Пользователь находит бота в MAX, нажимает «Запустить»
2. Бот показывает дисклеймер о ПД и просит согласие
3. После согласия — главное меню с каталогом
4. Пользователь выбирает мероприятие → видит карточку → нажимает «Записаться»
5. Бот создаёт запись и присылает:
   - **Оффлайн:** QR-картинку + код записи + инструкцию «покажите на входе»
   - **Онлайн:** ссылку для подключения + кнопку «🔗 Перейти к мероприятию»

### Сценарий «Контролёр → Допуск на входе»

1. Контролёр заходит в `/admin/login` со своего смартфона
2. На дашборде видит только назначенные ему мероприятия
3. Открывает нужное → кнопка «QR-сканер» → даёт доступ к камере
4. Сканирует QR абитуриента → мгновенно видит:
   - ✅ Допущен · Имя · Код
   - ❌ Уже отмечен / QR от другого мероприятия / отменено

### Сценарий «Организатор → Push-рассылка»

1. На карточке мероприятия — секция «Push-уведомление участникам»
2. Выбирается тип (информация / перенос / изменение места / напоминание 24ч / напоминание 1ч)
3. Текст уходит **только тем**, у кого не отключены уведомления по этой записи
4. История рассылок сохраняется со счётчиками доставленных/неуспешных

### Сценарий «Организатор → Назначение контролёров»

1. На карточке мероприятия — секция «Контролёры мероприятия»
2. Загрузка XLSX-файла со столбцами `email`, `full_name` (первая строка — заголовки)
3. Система ищет контролёров по email в БД:
   - **Найден** → привязывается к мероприятию
   - **Не найден** → в отчёте «не найдено», организатор обращается к техадмину для создания учётки

Принцип: **учётки создаёт только техадмин**, организатор только связывает существующие — это разделение ответственности.

## Безопасность и персональные данные

Принцип системы — **минимизация ПД** на всех уровнях.

### Что хранится про абитуриента
- `max_user_id` — внутренний идентификатор MAX (нужен, чтобы боту слать сообщения)
- `username` — отображаемое имя из профиля MAX
- Дата и версия согласия

**И всё.** Никаких email, телефонов, фамилий, дат рождения, образовательного учреждения. Если этого требует мероприятие — собирается отдельной формой за пределами системы.

### Защита данных в логах

Все логи проходят через `PIIRedactingFilter` (`app/logging_setup.py`):
- Email-адреса маскируются: `iv***@***`
- Длинные HMAC-токены: `iJxK7q…`
- Идентификаторы пользователей: `user:12…34`

Тело webhook-событий **никогда не пишется в лог** целиком — только тип и id отправителя.

### Аутентификация и сессии

- Пароли — `bcrypt` с cost=12 (~250 мс на хеш, выдерживает GPU-атаки)
- Сессии Flask — `HttpOnly` + `SameSite=Lax`
- Минимальная длина пароля — 6 символов (валидация на форме создания/редактирования)

### Проверка webhook от MAX

В подписке указан `secret` — MAX добавляет его в заголовок `X-Max-Bot-Api-Secret` к каждому запросу. Сервер проверяет через `hmac.compare_digest` (защита от timing-атак). Без правильного секрета — `401 Unauthorized`.

### QR-токены

Формат: `<reg_id>:<salt>:<hmac>`
- `reg_id` — id записи
- `salt` — случайные 8 байт (token_urlsafe)
- `hmac` — SHA-256 HMAC от `<reg_id>:<salt>` с секретом `QR_SECRET`

Без знания `QR_SECRET` подделать QR невозможно. Сервер при сканировании пересчитывает HMAC и сравнивает.

### Аудит-журнал

В `audit_log` пишутся все значимые действия:

| Действие | Кто инициатор |
|---|---|
| `consent_given` | абитуриент |
| `registration_created` / `registration_cancelled` | абитуриент |
| `organizer_login` / `admin_login` / `controller_login` | сотрудник |
| `event_created` / `event_updated` / `registration_closed` | организатор/админ |
| `attended_marked` | контролёр |
| `notification_sent` | организатор |
| `controllers_assigned` / `controller_revoked` | организатор |
| `user_created` / `user_deactivated` | техадмин |

В `payload` пишутся **только** id, типы и константы — никогда ПД.

Журнал доступен техадмину по `/admin/audit` — последние 200 событий с фильтрацией.

## Технические детали

### Адаптивная вёрстка

Все 8 шаблонов работают на экранах от 360px:
- **Гамбургер-меню** в шапке при ширине ≤ 768px
- **Таблицы → карточки** на мобильной: каждая ячейка получает подпись из `data-label` через `td::before { content: attr(data-label) }`
- **font-size: 16px** на всех input/select — отключает автоматический зум iOS Safari при фокусе
- **44px тап-зоны** на кнопках — соответствие Apple HIG и Material Design

### Логика QR vs ссылка для онлайн/оффлайн

```python
if event.format == EventFormat.online:
    # Текст со ссылкой + кнопка-link (если location — URL)
    send_link(...)
else:
    # QR-картинка + код записи
    send_qr(...)
```

Поле `location` в БД одно — для оффлайна это адрес, для онлайна URL встречи. Если строка начинается с `http://` или `https://` — на онлайн-мероприятии участнику показывается кнопка типа `link` для прямого перехода.

В записях `registration_code` и `qr_token` всё равно генерируются для онлайн-событий (целостность схемы), но пользователю не показываются.

### Загрузка картинок в MAX

`upload_image` в `services/max_client.py` делает два шага:
1. `POST /uploads?type=image` — получить upload URL
2. `POST <upload_url>` (multipart) — залить файл

Токен может прийти под разными ключами в ответе (`token`, `photo_token`, `file_token`, `id`, `file_id`, `fileId`, в `payload`, в `photos`). Функция `_extract_token` обходит структуру рекурсивно — это запас прочности на случай изменений в API MAX.

### Защита от частых сканирований

`scan.html` хранит последний токен и время скана. Один и тот же QR не отправится на сервер повторно в течение 3 секунд — это защита от случайного двойного срабатывания на физическом QR-листочке.

## Решение проблем

### Бот не отвечает в MAX
1. Проверьте, что webhook зарегистрирован
2. Сервер должен быть доступен по HTTPS из интернета. Локально — через ngrok.
3. `MAX_BOT_TOKEN` должен совпадать с тем, что у @MasterBot.

### Ошибка `psycopg2` при установке
На Windows — пакет уже бинарный (`psycopg2-binary`), не должен компилироваться. Если падает — проверьте, что Python из python.org, а не MSYS2.

### Меню в админке не открывается на мобильной
Сбросьте кэш браузера (`Ctrl+F5`) — могли остаться старые стили без гамбургер-кнопки.

### При входе постоянно «Неверный email или пароль»
- Учётка отключена техадмином (`is_active = false`)
- Опечатка в email (он case-insensitive, но пробелы важны)
- Проверьте `LOG_LEVEL=DEBUG` — в логах увидите детали попытки входа (без пароля)

### QR-сканер не запускается
Сканер использует API `getUserMedia` — нужен HTTPS (или `localhost`). На HTTP-странице камера в браузере недоступна по стандарту.

### Webhook возвращает 401
Не совпадает `WEBHOOK_SECRET` в `.env` и в зарегистрированной подписке. Перерегистрируйте webhook после смены секрета: `python -m migrations.register_webhook`.
